# CLYR Risk Register

Status: Phase 0 baseline with the Phase 1 implementation review recorded below. Most scanner, planning, execution, packaging, and release controls remain unimplemented.
Review cadence: at every phase exit, capability proposal, security-relevant dependency/rule change, packaging change, and incident/near miss.

## Rating and status rules

Risk ratings use only: `Informational`, `Low`, `Medium`, `High`, and `Prohibited`.

- `Informational`: observation or design dependency with no present material loss scenario; track for drift.
- `Low`: limited impact with tested controls and a clear owner.
- `Medium`: material product, privacy, reliability, or trust impact; requires a mitigation plan and owner before the affected phase ships.
- `High`: plausible severe data, privilege, privacy, or product-trust impact; release-blocking until evidence reduces it or explicit security/product acceptance is documented with an expiry.
- `Prohibited`: behavior violates a hard product boundary. It cannot be accepted, waived by the user, or shipped behind a flag.

Statuses are `Open`, `Mitigating`, `Accepted`, or `Closed`. `Accepted` records an informed residual-risk decision; it never applies to `Prohibited`. Since Phase 0 contains no executable cleanup, a design control is not marked `Closed` merely because it is documented.

Owners are accountable roles until maintainers are assigned by name. Every accepted risk needs rationale, approvers, evidence, review date, and expiry in the decision log.

## Risk acceptance procedure

1. The owner supplies reproducible evidence, affected capability/version/platform, current and residual rating, mitigations, alternatives, failure UX, and rollback/revocation plan.
2. `Medium` acceptance requires the accountable owner and phase maintainer. `High` acceptance additionally requires security and product-safety approval, a review expiry, and an entry in `docs/DECISION_LOG.md` or an ADR.
3. `Prohibited` is never accepted. Changing a hard boundary requires a new ADR that changes the architecture first; until approved and implemented, the existing prohibition remains.
4. Acceptance is invalidated by changed assumptions, platform/tool version, incident, failed regression test, or expiry.
5. Closed risks retain their history and evidence; they are reopened when a control or invariant regresses.

## Phase 0 exit criteria for this register

- Each identified severe data/privilege/privacy scenario has a canonical level, owner role, status, mitigation, residual statement, and phase gate.
- `Prohibited` entries have no waiver language or user override.
- No design-only control is represented as implemented or `Closed`.
- The threat, safety, privilege, IPC, error, ADR, and Mermaid documents use the same risk and action terminology.
- Phase reports list changes to this register and do not mark future execution risk resolved by documentation alone.

## Phase 0 closeout — 2026-07-12

The baseline register passed the cross-document terminology and ownership review. Documentation controls reduced design ambiguity for R-001 through R-029 but did not close any future implementation risk. `Prohibited` risks R-026 and R-027 remain non-waivable. R-028 (documentation/code drift) remains Open and is now backed by the local Phase 0 verifier, schema conformance, Mermaid rendering, and link/structure audits; Phase 1 must extend those checks into compiled architecture and safety tests. The packaging identity, stable file identity, journal durability, quarantine, adapter, and retention decisions above remain open gates for their stated phases.
